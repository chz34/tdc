# v3 — Inductor cpp_wrapper Probe (forward-only)

Status: draft, awaiting implementation
Owner: chz34
Date: 2026-05-28

## 1. Goal

Add a third capture path, `tdcv3`, to `torch_dispatch_capture` whose
sole purpose is to **probe and benchmark Inductor's
`cpp_wrapper`-based execution path** as a peer to v1 (dispatcher
fallback capture) and v2 (Dynamo + AOTAutograd → C++ Trace
translation).

The motivating question: how does v2's "boxed `callBoxed` over a
manually-translated trace" compare to Inductor's "unboxed C-shim
inside a cpp_wrapper-emitted C++ host driver" on the same workload,
once we factor in (a) per-call dispatch overhead, (b) recompile
overhead under shape variation, and (c) total capture cost?

v3 is a **thin adapter** around `torch.compile(backend="inductor",
dynamic=True)` plus `inductor_config.cpp_wrapper=True`. It is not a
new code generator. The value lives in the benchmark harness and the
correctness/diagnostics that make the comparison fair.

## 2. Non-goals

- **No backward.** v0.1 is forward-only. backward will be a separate
  follow-up if forward results warrant it. The benchmark table simply
  omits the v3 columns for backward scenarios.
- **No new C++ in this PR.** All code lives in Python under
  `python/v3/`. We do not add capture-side C++ to `csrc/`.
- **No persistent on-disk cache work.** We measure what Inductor's
  built-in caches give us; we do not augment them.
- **Not a drop-in for v2.** Different ergonomic constraints (e.g., v3
  inherits Dynamo's recompile-on-shape-change behaviour unless
  `dynamic=True` covers the spread). Users still pick v2 for the
  zero-recompile property.

## 3. User-facing API

```python
import torch_dispatch_capture.v3 as tdcv3

# Stock variant: only cpp_wrapper, Inductor still fuses & generates Triton/C++ kernels.
captured = tdcv3.capture(fn, *example_args, **example_kwargs)

# Fallback variant: cpp_wrapper + every op forced through aten via FallbackKernel.
# Mirrors v2's "no fusion, aten-only" execution shape.
captured = tdcv3.capture_fallback(fn, *example_args, **example_kwargs)

out = captured(*new_args, **new_kwargs)

# Diagnostics for the just-captured callable
report = tdcv3.last_capture_report()
# {
#   "variant": "fallback" | "stock",
#   "so_path": "/tmp/torchinductor_.../cuxs....so",
#   "cpp_source_path": ".../cuxs....cpp",
#   "fx_node_count": 47,
#   "fallback_node_count": 47,           # equals fx_node_count when variant=="fallback"
#   "capture_seconds": 2.14,
#   "recompile_count_after_capture": 0,
# }
```

Calling convention matches v2: positional + kwargs are passed through
unchanged; SymInt dimensions are inferred from `example_args` and the
returned callable accepts any compatible shape under `dynamic=True`.

## 4. Module layout

```
python/v3/
  __init__.py             # re-export capture, capture_fallback, last_capture_report
  compile.py              # capture() / capture_fallback() entry; calls torch.compile,
                          # primes the cache by invoking with example_args once,
                          # returns the compiled callable.
  fallback_hijack.py      # force_all_fallback() context manager.
test/
  test_v3_capture.py      # 6 tests (see §8).
prototypes/
  v3_benchmark.py         # Scenarios A/B/C, focused output (Markdown).
```

Edits to existing files:

- `prototypes/v2_benchmark.py`: extend `build_variants()` with two extra
  entries (`v3-stock`, `v3-fallback`) so the existing workload matrix
  picks v3 up for free. Variants are skipped (None entry) when v3
  capture fails on a workload — same pattern used today for inductor
  on NPU.
- `README.md`: short section explaining v3 alongside v1/v2.

No changes to `csrc/`, no changes to v1/v2 Python.

## 5. Implementation details

### 5.1 `compile.py`

```python
import contextlib
import time
from typing import Any, Callable, Tuple

import torch
import torch._inductor.config as inductor_config

from .fallback_hijack import force_all_fallback

_LAST_REPORT: dict | None = None
_LAST_GRAPH_STATS: dict = {}    # populated by the pre_grad_pass registered in §5.3


def capture(fn: Callable, *example_args, **example_kwargs) -> Callable:
    return _capture_common(fn, example_args, example_kwargs, fallback=False)


def capture_fallback(fn: Callable, *example_args, **example_kwargs) -> Callable:
    return _capture_common(fn, example_args, example_kwargs, fallback=True)


def last_capture_report() -> dict | None:
    return dict(_LAST_REPORT) if _LAST_REPORT is not None else None


def _capture_common(fn, example_args, example_kwargs, fallback: bool):
    global _LAST_REPORT
    base_recompile_count = _get_recompile_count()
    cm = force_all_fallback() if fallback else _stock_cpp_wrapper_config()

    t0 = time.perf_counter()
    with cm:
        compiled = torch.compile(fn, backend="inductor", dynamic=True)
        # Prime the cache so the user's first call is hot.
        _ = compiled(*example_args, **example_kwargs)
    t1 = time.perf_counter()

    _LAST_REPORT = _build_report(
        variant="fallback" if fallback else "stock",
        capture_seconds=t1 - t0,
        baseline_recompile_count=base_recompile_count,
    )
    return compiled


def _stock_cpp_wrapper_config():
    return inductor_config.patch({
        "cpp_wrapper": True,
        "triton.cudagraphs": False,   # cpp_wrapper × cudagraphs is incompatible
    })


def _get_recompile_count() -> int:
    return int(torch._dynamo.utils.counters["stats"].get("calls_captured", 0))


def _build_report(variant, capture_seconds, baseline_recompile_count):
    # Pulls the .so / .cpp path off the most-recent Inductor-emitted module via
    # torch._inductor.codecache.PyCodeCache global file list (best-effort).
    # If unavailable (very old / patched PyTorch), we leave the fields as None.
    so_path, cpp_path = _resolve_last_inductor_artifact_paths()
    return {
        "variant": variant,
        "so_path": so_path,
        "cpp_source_path": cpp_path,
        "fx_node_count": _LAST_GRAPH_STATS.get("fx_nodes"),
        "fallback_node_count": _LAST_GRAPH_STATS.get("fallback_nodes"),
        "capture_seconds": capture_seconds,
        "recompile_count_after_capture":
            _get_recompile_count() - baseline_recompile_count,
    }
```

`_LAST_GRAPH_STATS` is populated by a tiny custom `joint_graph_pass` or
by a `pre_grad_pass` registered for the duration of capture: it counts
the nodes in the FX graph it sees and, for the fallback variant,
double-counts after the lowering hijack runs. Implementation detail —
see §5.3 for the exact hook.

### 5.2 `fallback_hijack.py`

```python
import contextlib
import torch
import torch._inductor.config as inductor_config
from torch._inductor.lowering import lowerings, fallback_handler


@contextlib.contextmanager
def force_all_fallback():
    """Temporarily replace every OpOverload lowering with its fallback handler.

    Combined with `cpp_wrapper=True` and fusion-disabling flags, this makes
    Inductor act as: "C-shim host wrapper that calls aten kernels in
    sequence" — the form most directly comparable to v2.

    NOT thread-safe: monkey-patches the module-level lowerings dict.
    Acceptable for single-process benchmarking; v3 documents this.
    """
    saved = dict(lowerings)
    try:
        for key in list(lowerings.keys()):
            if isinstance(key, torch._ops.OpOverload):
                lowerings[key] = fallback_handler(key, add_to_fallback_set=False)
            # OpOverloadPacket / str keys: leave alone. Packets are
            # resolved per-overload at lookup time; strs are prim
            # decompositions whose handlers are themselves Python and
            # routed through OpOverload entries that we *do* patch.
        with inductor_config.patch({
            "cpp_wrapper": True,
            "epilogue_fusion": False,
            "max_fusion_size": 1,
            "triton.cudagraphs": False,
            "freezing": False,
        }):
            yield
    finally:
        lowerings.clear()
        lowerings.update(saved)
```

### 5.3 Graph-stats hook

To populate `fx_node_count` and `fallback_node_count`, register a
`pre_grad_pass` for the duration of capture using
`inductor_config.patch({"pre_grad_custom_pass": _stats_pass})`. The
pass walks the FX graph and counts nodes whose target is an
`OpOverload`; it returns the graph unchanged.

For `fallback_node_count`, register a similarly-scoped `joint_graph_pass`
that runs AFTER lowering decisions are observable — concretely, count
`FallbackKernel` nodes via Inductor's IR observation hook
(`V.graph.scheduler.nodes` post-lowering). If the hook surface proves
fragile across PyTorch versions, degrade gracefully: report
`fallback_node_count = fx_node_count` when `variant=="fallback"` (the
hijack guarantees this anyway, the count is just for sanity).

### 5.4 Capture-cost measurement

`capture_seconds` is wall-clock around the `torch.compile + primer
call`. Includes Dynamo trace + AOT + Inductor lowering + C++ compile +
dlopen — i.e., the total "first-call latency" the user sees. This is
the apples-to-apples number to put against v2's `capture(...)`
wall-clock.

### 5.5 Diagnostic artifact-path resolution

`_resolve_last_inductor_artifact_paths()`:

- Walks `torch._inductor.codecache.PyCodeCache.cache` for entries
  added since `t0`.
- For each, the value is a module whose `__file__` is the emitted
  `.cpp`-shaped Python wrapper (containing the C++ source as a raw
  string). The matching `.so` lives next to it under the
  `torchinductor` cache root.
- If exact identification fails (multiple new entries from
  concurrent autotune), we record the path of the largest new file
  (the cpp_wrapper output dominates triton kernel artifacts in size).
  Best-effort by design — this is diagnostics, not load-bearing.

## 6. Benchmark plan

`prototypes/v3_benchmark.py`. Five variants × three scenarios.

### 6.1 Variants

| Variant       | Wrapper          | Kernel codegen       | Notes                                              |
| ------------- | ---------------- | -------------------- | -------------------------------------------------- |
| `eager`       | –                | –                    | baseline                                           |
| `inductor`    | Python           | fused Triton/C++     | Stock `torch.compile(backend="inductor")`          |
| `v3-stock`    | C++ (cpp_wrapper)| fused Triton/C++     | Isolates *wrapper-shape* contribution             |
| `v3-fallback` | C++ (cpp_wrapper)| 100% aten fallback   | Most-like-v2 form                                  |
| `v2`          | C++ (Trace)      | 100% aten            | boxed `callBoxed` + `AutoDispatchBelowAutograd`    |

`v1` is excluded from v3 benchmarks — forward-only KV-cache-style
workloads where v1 dominates are already covered in
`v2_benchmark.py`.

### 6.2 Scenarios

**Scenario A — fixed-shape micro-op hot loop.**
Reuse `workload_pointwise(x, y)`, `workload_tiny(x)`, `workload_attention(q, k)`
from `v2_benchmark.py`. Inputs sized to fit in L2 to avoid memory-bound
noise (e.g., `B=4, dim=256`).

Procedure: 100 warmup + 1000 timed calls.
Metric: `per_call_us` (median).
Expectation: `v3-stock ≈ inductor < v3-fallback < v2 ≈ eager`. The gap
between `v3-fallback` and `v2` quantifies *unboxed C-shim vs boxed
`callBoxed`*; the gap between `v3-stock` and `inductor` quantifies
*C++ wrapper vs Python wrapper* (likely small for hot loops where
Python overhead is amortised by torch.compile guards).

**Scenario B — dynamic shape (seq_len varies).**
A fixed GEMM + pointwise chain, run sequentially with
`seq_len ∈ {128, 256, 512, 1024}`. Each shape: 50 calls.

`v3-stock` / `v3-fallback` are captured with `dynamic=True`; we also
capture `torch._dynamo.utils.counters["stats"]["calls_captured"]`
deltas to verify recompile count.

Metrics per shape:
- `first_call_ms` (includes any recompile)
- `steady_state_us` (calls 2..50 median)
- `recompiles` total over the four shapes

Expectation: v2 has 0 recompiles after the initial capture. v3 *should*
also have 0 recompiles given `dynamic=True`, but real-world Dynamo
guards may force one (esp. on shape-dependent control flow); the
benchmark exists to find out.

**Scenario C — Transformer Block / Llama Attention.**
Reuse `_build_transformer_block(hidden=512, n_heads=8, ffn_inner=2048)`
and `_build_llama_attention(dim=512, n_heads=8, max_batch=2, max_seq=128)`.
Fixed shape, 100+1000 timing.

Metrics:
- `per_call_us`
- `max_abs_diff_vs_eager` (numerical check, atol/rtol 1e-3/1e-3 must pass)

## 7. Failure modes & mitigations

| Failure                                                                 | Mitigation                                                                                                                                                                                          |
| ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Some op has no `FallbackKernel` (no functional schema)                  | Catch `LoweringException` in capture, surface a clear "v3-fallback not supported for op X" error pointing to the workload. Document known incompatibles in README.                                  |
| `cpp_wrapper=True` + cudagraphs implicit-on                             | The patch in `force_all_fallback` / `_stock_cpp_wrapper_config` explicitly sets `triton.cudagraphs=False`. We also log a warning if `config.triton.cudagraphs` was True at entry.                    |
| Concurrent test execution mutates `lowerings`                           | Document that `force_all_fallback` is single-threaded. Tests use `unittest.TestCase` (serial by default). Add an `assert threading.active_count() == 1` guard at context entry, with a kill-switch. |
| Inductor codecache layout changes between PyTorch versions              | `_resolve_last_inductor_artifact_paths()` is best-effort and gracefully reports `None`. The benchmark does not depend on these paths.                                                               |
| `dynamic=True` still triggers recompile on Scenario B                   | This is a *finding*, not a bug. The benchmark prints `recompiles` per variant so the reader sees the truth.                                                                                          |
| Numerical drift between v3-stock (fused Triton) and eager beyond 1e-3   | Loosen tolerances per-scenario in the benchmark (warn, don't fail); tighten in the test suite where exactness matters.                                                                              |

## 8. Tests (`test/test_v3_capture.py`)

All tests follow the v2 test file conventions: `TestCase` per concern,
read `TDC_DEVICE`, use `tdcv3.capture` / `capture_fallback`. The suite:

1. **`test_smoke_stock`** — `tdcv3.capture(fn)` on a tiny pointwise fn;
   call result `allclose` eager within `atol=1e-3`.
2. **`test_smoke_fallback`** — same fn through `tdcv3.capture_fallback`;
   `allclose` eager within `atol=1e-3` (AOT decomposition still applies
   even with all-fallback lowering, so layer_norm / softmax / etc.
   reorder fp32 FMAs at the same level v2 does).
3. **`test_fallback_really_all_fallback`** — capture an attention-like
   block; assert `report["fallback_node_count"] == report["fx_node_count"]`
   AND assert (via Inductor IR introspection) that every op node in the
   compiled graph has `_is_fallback_handler == True`.
4. **`test_dynamic_shape_capture_once`** — single `capture_fallback` call,
   then invoke with three different shapes; assert
   `report["recompile_count_after_capture"] == 0` AND the per-shape
   results match eager.
5. **`test_kwargs_passthrough`** — capture a fn with one positional +
   one keyword arg; call by mixing positional/keyword forms.
6. **`test_capture_report_paths`** — verify `report["so_path"]` exists
   on disk when capture succeeds; `None` when capture failed gracefully.

The test file declares dependencies on `torch._inductor.lowering`
internals (the lowerings hijack); if those imports fail in a stripped
PyTorch build, the whole file skips with a clear message.

## 9. Out of scope (deferred)

- **v3 backward** (`allow_grad=True`). Requires understanding how
  AOTAutograd's RuntimeWrapper interacts with `cpp_wrapper` mode for
  the backward subgraph; this is a non-trivial integration.
- **AOTInductor (.so deploy) comparison.** Different deployment story;
  belongs in its own analysis.
- **Comparison against `torch.compile(backend="inductor",
  mode="reduce-overhead")`** (which uses cudagraphs). Excluded because
  cudagraphs is incompatible with cpp_wrapper.
- **Multi-process / thread-safe `force_all_fallback`.** Single-process
  benchmark is the design constraint.
